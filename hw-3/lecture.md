### **1. Основные дизайн-паттерны в DeFi**

*   **Deployer (Create2 / Create3):**
    *   Позволяет получать **детерминированные адреса** контрактов в разных сетях.
    *   Используется для создания однотипных контрактов с контролем доступа (роли `deployer` и `approver`).
    *   Пример: Протокол **1inch** использует `Create2` для генерации адресов с набором единиц в начале как доказательство валидности контракта.
    *   `Create3` - обёртка (например, в деплоерах типа Solady), избавляющая от зависимости от кода инициализации.

*   **Factory (Фабрика):**
    *   Создает множество инстансов контрактов с идентичной логикой (токены, пулы, валты).
    *   Используется в DEX (например, Uniswap) для создания торговых пар.
    *   Функция `create` принимает аргументы, которые передаются напрямую в конструктор целевого контракта.

*   **Clones / Minimal Proxy (Клоны):**
    *   Создают минимальную прокси-оболочку вместо деплоя полного кода имплементации, что значительно **снижает газовые расходы** (в 10–50 раз).
    *   Пример: **Gnosis Safe**, где каждый кошелек - это минимальный клон.

*   **Adapter (Адаптер):**
    *   Нормализует внешние протоколы под единый локальный интерфейс.
    *   Критически важен для **DEX-агрегаторов**, позволяя работать с Uniswap V2, V3, Curve через универсальный интерфейс.
    *   Снижает затраты на аудит, так как бизнес-логика работает с абстракциями.

*   **Registry (Реестр):**
    *   Централизованное хранилище данных (конфигурации, риск-метрики, адреса контрактов).
    *   Позволяет избежать хардкода и упрощает управление системой через единый конфигуратор.

*   **Router / Multicall (Роутер):**
    *   Перенаправляет логику выполнения и объединяет несколько действий (например, `approve` + `swap`) в одну транзакцию.
    *   Используется в Uniswap Universal Router для композиции сложных сделок.

### **2. Антипаттерны разработки**

*   **God Contract (Божественный контракт):** Весь функционал в одном месте. Ведет к проблемам с обновляемостью, лимитам размера контракта в EVM и является единой точкой отказа.
*   **Root User:** Наличие одного суперпользователя с доступом ко всему. Обязанности и доступы должны быть разделены.
*   **Proxy Initialization:** Оставление имплементации прокси неинициализированной открывает векторы атак.
*   **Unchecked external calls:** Отсутствие проверок и порогов (thresholds) при изменении критических параметров (комиссии, LTV).

### **3. Эволюция архитектуры Uniswap**

*   **V2:** Классическая схема с фабрикой пулов (прямые имплементации, не клоны) и роутером.
*   **V3:** Усложнение архитектуры. Появление **Non-fungible Position Manager** (ликвидность теперь в виде NFT), библиотек для чтения данных (Quoter, Lens).
*   **V4:** Кардинальная смена парадигмы. 
    *   **Singleton (PoolManager):** Все пулы находятся в одном контракте.
    *   **Transient Storage (`tstore`):** Использование временной памяти внутри транзакции ("flash accounting") для снижения стоимости газа.
    *   **Hooks (Хуки):** Возможность выполнять кастомный код до/после свапа или изменения ликвидности.

### **4. Инфраструктура: Оракулы (Chainlink)**

Для работы финансовых систем (лендингов) необходимы данные из внешнего мира, которыми нельзя манипулировать в рамках блока.
*   **Price Feeds:** Децентрализованные агрегаторы цен (медиана от разных нод).
*   **CCIP:** Протокол для кроссчейн-взаимодействия.
*   **Automation & VRF:** Сервисы для запуска контрактов по условию (Cron) и генерации верифицируемой случайности.

### **5. Практическая часть: Реализация прокси (UUPS)**

На семинаре продемонстрирован процесс создания обновляемого контракта:
*   **CounterV1 и CounterV2:** Реализация логики с использованием `Initializable` и `UUPSUpgradeable` от OpenZeppelin.
*   **Inherited Storage:** Паттерн наследования стореджа, где новые переменные (например, `step` в V2) добавляются в новые слоты.
*   **Особенности:**
    *   Вместо конструктора используется функция `initialize` с модификаторами `initializer` (V1) или `reinitializer` (V2).
    *   Функция `_authorizeUpgrade` ограничивает право на обновление контракта (только владелец/апгрейдер).
*   **Тестирование и скрипты:** Показан деплой через Foundry, выполнение апгрейда командой `upgradeToAndCall` и проверка сохранения состояния между версиями.

### **6. Практическая часть: Factory и Registry**
Создана простая фабрика кошельков (`SafeFactory`), которая при создании нового объекта записывает его адрес в массив `contractsRegistry`. Это позволяет пользователям находить свои контракты через единую точку доступа.
